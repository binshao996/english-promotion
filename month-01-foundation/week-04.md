## 1. Grammar: Expressing Future（表达将来的四种方式）

中文表达将来很简单——加个"将"、"会"、"打算"、"要"就行，动词本身不变。但英语有四种完全不同的结构来表达将来，选哪一种取决于你对这件事的"确定程度"和"计划程度"。选错了，意思会有微妙但重要的差别。

---

### Part A: will（将 / 会）

**核心概念：** will 是英语中最"灵活"的将来表达方式。它不强调"计划"，而强调"预测"、"当场决定"或"承诺"。

#### 1. 核心用法

**用法 1：当场做出的决定**
在说话的这一刻才决定要做某事——不是提前计划好的。
- "The build just failed — I'll check the logs." 构建刚刚失败了——我去看看日志。
- "That's a good point. I'll add error handling for that case." 好主意，我给那个情况加上错误处理。
- "The meeting room is busy? OK, I'll book a different room." 会议室有人？行，那我订另一间。

**用法 2：基于主观判断的预测**
你认为某事会发生，但这不是基于具体的证据，而是基于你的经验或直觉。
- "I think the new architecture **will** scale to 10x the current traffic." 我认为新架构能扩展到当前流量的 10 倍。
- "The migration **will** probably take about 4 hours." 迁移大概需要四小时。
- "I'm sure the team **will** love the new monitoring dashboard." 我确定团队会喜欢新的监控仪表盘。

**用法 3：承诺 / 提议 / 主动帮忙**
明确表示你会做某事——这是 will 最重要的社交功能之一。
- "I'll send you the meeting notes after the call." 我会在会后把会议记录发给你。
- "I'll investigate the root cause and report back." 我会调查根本原因并回报。
- "Don't worry — I'll handle the deployment tonight." 别担心——今晚我来部署。

**用法 4：关于未来的客观事实**
描述一定会发生的未来事件——不受人的意愿影响。
- "The certificate **will** expire on June 30th." 证书将在6月30日过期。
- "The trial period **will** end automatically after 30 days." 试用期将在30天后自动结束。
- "The database **will** reach its storage limit in about two weeks at the current growth rate." 按照当前的增长速度，数据库大约两周后达到存储上限。

#### 2. 形式

**肯定：** will + 动词原形（will 永远不会变成 wills，动词永远是原形）
- I **will deploy** → I'll deploy
- She **will review** → She'll review
- The server **will restart** → The server'll restart（口语中少见，通常说全称）

**否定：** will not = won't（不是 willn't！）
- "I **won't** merge without a review."
- "The bug **won't** be fixed by tomorrow."

**疑问：** Will + 主语 + 动词原形 + ?
- "**Will** you **attend** the incident review meeting?"
- "**Will** the hotfix **be ready** by tonight?"

#### 3. 完整语法拆解示例

```
示例 1：The build just failed — I'll check the logs.
- The build (主语/名词短语) + just failed (谓语/一般过去时) — I'll (主语 I + will 缩写)
  + check (谓语/动词原形) + the logs (宾语/名词短语)
- 分析：前半句是现场情况描述（build just failed），后半句是当场做出的决定（I'll check）。
  "just" 表示"刚刚"。"I'll" = I will + 动词原形 check。
- 理解技巧：你不是提前计划好要检查日志的，是看到 build 失败后当场决定的。
  这种"即时决定"只能用 will，不能用 be going to。
- 常见错误：用 be going to 来表达当场决定。
  ❌ "The build failed — I'm going to check the logs."
  ✅ "The build failed — I'll check the logs."
  （be going to 暗示你之前已经计划好了，但这里是当场决定。）
```

```
示例 2：I will investigate the root cause and report back by EOD.
- I (主语/代词) + will (情态动词表将来) + investigate (谓语/动词原形)
  + the root cause (宾语/名词短语) + and (连接词) + report back (谓语 + 副词)
  + by EOD (时间状语)
- 分析：will + 动词原形 investigate。EOD = End of Day，是工作中常见的缩写。
  "and report back" 连接两个并列动词（investigate 和 report），都跟在 will 后面，
  所以都是原形。
- 理解技巧：这是一个承诺（promise）。你用 will 向对方保证你会做这件事。
  "by EOD" 是截止时间，放在句末。
- 常见错误：will 后面加 -ing 或 -s。
  ❌ "I will investigates the root cause."（will 后面不能加 -s）
  ❌ "I will investigating the root cause."（will 后面用原形，不是 V-ing）
  ✅ "I will **investigate** the root cause."
```

```
示例 3：The certificate will expire on June 30th.
- The certificate (主语/名词短语) + will (情态动词) + expire (谓语/动词原形)
  + on June 30th (时间状语)
- 分析：主语是第三人称单数 "The certificate"，但 will **不变化**。
  "will expire" 在语法上是"一般将来时"结构。
- 理解技巧：这是"关于未来的事实"——证书过期是一个确定的事件，
  不受人的主观意志影响。不管你想不想，它都会过期。
- 常见错误：
  ❌ "The certificate wills expire."（will 不能加 -s）
  ❌ "The certificate will expires."（expire 不能加 -s）
  ✅ "The certificate will expire."（will + 原形，干干净净）
```

```
示例 4：I won't merge the PR until the tests pass.
- I (主语/代词) + won't (will not 缩写) + merge (谓语/动词原形)
  + the PR (宾语/名词短语) + until the tests pass (时间状语从句)
- 分析：won't = will not，放在主语和主要动词之间。until 从句用一般现在时
  "the tests pass" 表将来——这是英语的固定规则：时间状语从句中将来时用
  一般现在时代替。
- 理解技巧：won't = 将来不会做某事。"until" 表示"直到……为止"。
  从句用一般现在时（pass）而不是 will pass，这是中国学生最容易忽略的规则之一。
- 常见错误：在 until/if/when 从句中用 will。
  ❌ "I won't merge until the tests will pass."（时间状语从句中不用 will）
  ✅ "I won't merge until the tests pass."（用一般现在时代替将来）
```

### Part B: be going to（打算 / 将要）

**核心概念：** be going to 强调的是"计划"和"预兆"——这件事不是"我要不要做"的问题，而是"已经决定要做"或"显然会发生"。

#### 1. 核心用法

**用法 1：预先计划的意图/intention**
在说话之前就已经决定好了要做某事——它不是一个临时决定。
- "We're going to migrate the database this weekend." 我们打算这周末迁移数据库。
- "I'm going to refactor the auth module next sprint." 我计划下个 sprint 重构认证模块。
- "She's going to propose a new CI pipeline architecture at the tech meeting." 她打算在技术会议上提议新的 CI 管道架构。

**用法 2：基于当前证据的预测**
你"看到"了一些迹象，由此推断某事会发生——不是主观猜测，而是有依据的推断。
- "Look at these error rates — the server **is going to crash**." 看看这些错误率——服务器要崩溃了。
- "The disk usage is at 95%. It's going to run out of space by tonight." 磁盘用量到 95% 了，到今晚就会跑满。
- "With all these new features being added, the build time **is going to increase** significantly." 加了这么多新功能，构建时间会显著增加。

#### 2. 形式

**肯定：** am/is/are + going to + 动词原形
- I **am going to** rewrite the module.
- He/She/It **is going to** approve the PR.
- We/You/They **are going to** deploy on Friday.

**否定：** am/is/are + not + going to + 动词原形
- "I'm **not going to** approve this PR until the tests pass."
- "The team **isn't going to** meet the deadline."

**疑问：** Am/Is/Are + 主语 + going to + 动词原形 + ?
- "**Are** you **going to** attend the incident review?"
- "**Is** the server **going to** handle the traffic spike?"

#### 3. 完整语法拆解示例

```
示例 1：We're going to migrate the database this weekend.
- We (主语/代词) + 're going to (are going to 缩写) + migrate (谓语/动词原形)
  + the database (宾语/名词短语) + this weekend (时间状语)
- 分析：be (are) + going to + 动词原形。这是"预先计划"的用法——
  迁移数据库不是临时起意，而是已经计划好的周末操作。
- 理解技巧：如果用 will 说 "We'll migrate the database this weekend"，
  听起来像"好吧，那就这周末移吧"（当场决定）。用 be going to 则说明
  "这早就在计划中了"。
- 常见错误：
  ❌ "We going to migrate the database."（缺了 are）
  ✅ "We **are** going to migrate the database."
  ❌ "We are going to migrate the database this weekend, we planned it last month."
  （这句话是对的，但 go to 和 going to 连在一起不好读——不如直接用
  "We planned to migrate..." 更清晰）
```

```
示例 2：The disk is at 95% — it's going to run out of space soon.
- The disk (主语/名词短语) + is at 95% (系表结构) — it (主语/代词)
  + is going to (表将来) + run out of (动词短语) + space (宾语)
  + soon (时间状语/副词)
- 分析：前半句是"当前证据"（磁盘 95%），后半句是基于证据的预测（即将耗尽）。
  这是 be going to 区别于 will 最重要的特征——你能"看到"迹象。
- 理解技巧：will 预测 = "我觉得可能会……"（主观），be going to 预测 =
  "看这情况，肯定要……"（有证据）。如果你看到磁盘快满了，
  说 "It's going to crash" 比 "It will crash" 更自然，因为你有证据。
```

```
示例 3：I'm going to propose a new monitoring strategy in the next retro.
- I (主语/代词) + 'm going to (am going to 缩写) + propose (谓语/动词原形)
  + a new monitoring strategy (宾语/名词短语) + in the next retro (时间状语)
- 分析：这是一个"计划中的行动"——不是临时在 retro 上说的，
  而是你在 retro 之前就已经准备要提的建议。
- 理解技巧：be going to 暗示"我已经在准备了"。
  如果我说 "I'm going to propose..."，我的同事会理解为我已经想好了要说什么。
  如果我说 "I'll propose..."，听起来像"那行，那我在 retro 上说一下"。
- 常见错误：
  ❌ "I'm going to propose a new monitoring strategy, I prepared the slides."
  （两个句子用逗号连接——这是逗号拼接/ comma splice 错误）
  ✅ "I'm going to propose a new monitoring strategy **.** I **already** prepared the slides."
```

### Part C: Present Continuous for Future（现在进行时表将来）

**核心概念：** 现在进行时表将来强调的是"已经安排好的约会/会议/行程"。这是英语口语中最常见的将来表达方式——比 will 更常用。

#### 1. 核心用法

必须包含一个将来的时间标志词！没有时间词，听起来就是"现在正在做"。

- "I'm meeting the client tomorrow at 3 PM." 我明天下午3点和客户见面。
- "We're deploying the hotfix tonight." 我们今晚部署热修复。
- "He's presenting the architecture proposal on Friday." 他周五做架构方案演示。

注意事项：现在进行时表将来只能用于"人的安排"——不能用于"自然会发生的事"。
- ❌ "It's raining tomorrow."（下雨不是人能安排的）
- ✅ "It's going to rain tomorrow."（用 be going to 表示预测）
- ✅ "I'm flying to Shanghai next week."（飞行是人安排的，可以）

#### 2. 完整语法拆解示例

```
示例 1：I'm meeting the product manager tomorrow to discuss the roadmap.
- I (主语/代词) + 'm meeting (am meeting 缩写/现在进行时表将来)
  + the product manager (宾语/名词短语) + tomorrow (时间状语/将来标志词)
  + to discuss the roadmap (不定式/目的状语)
- 分析：现在进行时 am + meeting，但 tomorrow 明确表示这是将来安排。
  没有 tomorrow → "I'm meeting the product manager" 听起来像"此刻正在开会"。
- 理解技巧：现在进行时表将来 = "我日历上有这项安排"。
  用 will 说 "I'll meet the PM tomorrow" 听起来不自然——因为 meeting 通常是提前约好的，
  不是当场决定的。安排好的事，用现在进行时。
- 常见错误：忘记加时间标志词。
  ❌ "I'm meeting the client."（是现在还是将来？不清楚）
  ✅ "I'm meeting the client **on Thursday**."（时间词明确了这是将来安排）
```

```
示例 2：We're deploying the security patch tonight at midnight.
- We (主语/代词) + 're deploying (are deploying/现在进行时)
  + the security patch (宾语/名词短语) + tonight at midnight (时间状语)
- 分析：现在进行时 are + deploying，但 tonight at midnight 明确是将来时间。
- 理解技巧：这句话隐含的意思是"部署已经安排好了——发布时间、人员、步骤都已确定"。
- 常见错误：在用现在进行时表将来时，把动词改成 will 看起来很"正式"但不自然。
  ❌ "We will deploy the security patch tonight at midnight."
  （语法没错，但听起来像"好，那就今晚部署吧"——现场决定的感觉）
  ✅ "We're deploying the security patch tonight at midnight."
  （自然的表达——这件事已经安排好了）
```

```
示例 3：He's flying to Singapore on Monday for the conference.
- He (主语/代词) + 's flying (is flying/现在进行时表将来)
  + to Singapore (地点状语) + on Monday (时间状语) + for the conference (目的状语)
- 分析：现在进行时 is + flying。用现在进行时表将来的动词通常是"移动类"动词——
  go, come, leave, arrive, fly, drive, travel 等。
  "on Monday" 是具体时间——用 on 表示具体某一天。
- 理解技巧：移动类动词 + 现在进行时是英语中最常见的"将来"表达。
  "I'm leaving"、"She's coming"、"We're going"——这些听起来都很自然。
  如果是 will："I will leave on Monday" 听起来更像"我决定周一走"。
- 常见错误：在时间词前用错介词。
  ❌ "He's flying to Singapore in Monday."（in 用于月份/年/时间段）
  ✅ "He's flying to Singapore **on** Monday."（on 用于具体某一天）
```

### Part D: Present Simple for Future（一般现在时表将来）

**核心概念：** 一般现在时表将来用于"固定时间表"——这些事件不依赖于人的意愿，是已经定死的日程。常见的动词只有一小类：start, end, begin, finish, open, close, arrive, depart, leave 等。

#### 1. 核心用法

- "The sprint **ends** on Friday." 这个 sprint 周五结束。
- "The conference **starts** at 9 AM and **finishes** at 5 PM." 会议早上9点开始，下午5点结束。
- "The release window **opens** on Monday and **closes** on Wednesday." 发布窗口周一开始，周三关闭。

注意：这种用法最常用于"官方时间表"——sprint 周期、release 计划、会议的起止时间、列车的发车时间等。

#### 2. 完整语法拆解示例

```
示例 1：The sprint ends on Friday, so we need to wrap up our tasks.
- The sprint (主语/名词短语) + ends (谓语/一般现在时第三人称单数)
  + on Friday (时间状语) + so we need to wrap up our tasks (结果状语从句)
- 分析：sprint 的结束时间是团队在 sprint planning 时就定好的。
  即使现在是周三，"on Friday" 是将来时间，但仍然用一般现在时 ends。
- 理解技巧：一般现在时表将来 = "死时间，板上钉钉"。没有人能在 sprint 中途说
  "sprint 延长到下周"而不经过正式的重新规划。
- 常见错误：用 will 来表达已经确定的时间表。
  ❌ "The sprint will end on Friday."（语法对，但听起来不自然——好像你在预测，
  但 sprint 的结束时间是定好的）
  ✅ "The sprint **ends** on Friday."（自然的表达——这是固定时间表）
  （当然，"will end on Friday" 也可以，但 "ends on Friday" 更地道。）
```

```
示例 2：The deployment window opens at 10 PM and closes at 2 AM.
- The deployment window (主语/名词短语) + opens (谓语/一般现在时第三人称单数)
  + at 10 PM (时间状语) + and (连接词) + closes (谓语/第三人称单数)
  + at 2 AM (时间状语)
- 分析：两个动词 opens 和 closes 都是第三人称单数，因为主语是
  "The deployment window"。虽然说的是将来的事，但用的是"时间表"语法。
- 理解技巧：想象成"列车时刻表"——火车不会因为"还没到时间"就用将来时。
  火车时刻表上写的是 "The train departs at 8:30"，不是 "will depart"。
  部署窗口也一样，它有一个预定的起止时间。
```

```
示例 3：The new API version goes live next quarter.
- The new API version (主语/名词短语) + goes live (谓语/动词短语，go 加 -es)
  + next quarter (时间状语)
- 分析："goes live" = "上线/生效"。主语是第三人称单数，所以 go → goes。
- 理解技巧：这是公司路线图上的计划——已经确定好了的，不是"如果……就……"。
  像产品发布计划这种"时间表"，用一般现在时是很常见的。
- 常见错误：不要随意把任何动词都用在"一般现在时表将来"结构中。
  ❌ "I submit the report next week."（submit 不是"时间表"动词，听起来奇怪）
  ✅ "The deadline **is** next week. I'll submit the report by then."
  （用 be 动词 + 时间表，或者用 will 表达承诺）
  ✅ 时间表类才用：start, end, begin, finish, open, close, arrive, depart, go live
```

### Part E: Contrast Table（四种将来表达的对比）

| 表达方式 | 核心含义 | 何时使用 | 一句话场景对比 |
|---------|---------|---------|--------------|
| **will** | 将/会（当场决定/预测/承诺） | 说话时才做的决定；基于主观经验的预测；主动提建议或承诺 | PM 问谁能看看这个 bug——"I'll look into it."（当场决定） |
| **be going to** | 打算/将要（计划/预兆） | 提前计划好的事；有明显证据的预测 | 你已经在看 bug 了——"I'm going to fix it after lunch."（计划好的） |
| **Present Continuous** | 安排好了（时间地点已定） | 有具体时间地点的约会、会议、行程——日历上已经有条目了 | 你已经和 QA 约好了——"I'm debugging it with Sarah at 3 PM."（有日程安排） |
| **Present Simple** | 固定时间表（板上钉钉） | 官方时间表、固定起止时间——不受个人意志影响 | 这是 sprint 的截止时间——"The sprint ends on Friday."（时间表固定） |

**场景深度对比 — 你在一场 incident review 会议上：**

| 你说的话 | 表达方式 | 潜台词 |
|---------|---------|--------|
| "I'll investigate the root cause." | will | 你是在会上当场做出的承诺——之前没想过这事 |
| "I'm going to investigate the root cause." | be going to | 你之前已经计划好了要查——可能已经在看了 |
| "I'm investigating the root cause with Tom tomorrow." | Present Continuous | 你和 Tom 已经约好了明天一起排查——日历有安排 |
| "The investigation deadline is next Friday." | Present Simple | 截止日期是上面定好的，不是你能改的 |

**更多横向对比：**

| 场景 | will | be going to | Present Continuous（将来） | Present Simple（将来） |
|------|------|------------|--------------------------|----------------------|
| 部署 | "I'll deploy it."（当场决定） | "I'm going to deploy tonight."（计划） | "I'm deploying at 10 PM."（安排好了） | "The deployment window opens at 10 PM."（时间表） |
| bug 修复 | "I'll fix that."（主动揽活） | "I'm going to fix this sprint."（列入计划） | "I'm fixing it tomorrow morning."（已安排） | "The hotfix goes live on Wednesday."（发布时间定好） |
| 会议 | "I'll attend."（当场答应） | "I'm going to attend."（决定要去） | "I'm attending the retro."（已约好） | "The retro starts at 10 AM."（时间固定） |

---

### Part F: Common Chinese-speaker Errors with Future Expressions

**错误 1：所有将来情况都用 will**
中文一个"会"/ "将"字走天下，但英语需要根据语境选择。
- ❌ "The sprint will end on Friday, so we need to hurry."（语法没错，但不自然）
- ✅ "The sprint **ends** on Friday, so we need to hurry."（时间表用一般现在时更地道）
- ❌ "I will meet the client tomorrow at 3 PM."（语法没错，但听起来像当场决定）
- ✅ "I'm meeting the client tomorrow at 3 PM."（安排好的事用现在进行时）
- **原则：** 先问自己"这件事是安排好的吗？→ 现在进行时。是时间表吗？→ 一般现在时。是计划吗？→ be going to。只有当场决定/预测/承诺才用 will。

**错误 2：will 后面加 -ing 或 -s**
- ❌ "It will works."（will 后面的动词不能加 -s）
- ❌ "It will working."（will 后面的动词不能加 -ing）
- ✅ "It **will work**."（will + 动词原形）
- 记住：will + 原形——will 像 should/can/must 一样，后面永远跟动词原形。

**错误 3：be going to 忘记 be 动词**
- ❌ "We going to migrate the database."（缺了 are）
- ✅ "We **are** going to migrate the database."
- 原因：中文说"我们打算迁移数据库"——没有"是"字。但英语的 be going to 必须有 be 动词。

**错误 4：时间状语从句中用了 will**
在 if, when, until, as soon as, before, after 引导的时间/条件状语从句中，即使表示将来，也不能用 will，要用一般现在时。
- ❌ "I'll deploy the patch when the tests **will pass**."
- ✅ "I'll deploy the patch when the tests **pass**."
- ❌ "If the server **will crash**, we'll lose data."
- ✅ "If the server **crashes**, we'll lose data."

**错误 5：现在进行时表将来忘了时间词**
- ❌ "I'm meeting the client."（是现在还是将来？说不清）
- ✅ "I'm meeting the client **tomorrow**."
- ❌ "We're deploying."（听起来像"正在部署"）
- ✅ "We're deploying **tonight**."
- 规则：没有将来时间词，现在进行时默认表示"现在"。

**错误 6：把 will not 缩写写成 willn't**
- ❌ "I willn't merge it."（willn't 不存在）
- ✅ "I **won't** merge it."
- 例外：英国口语中偶尔有 "I'll not" 的形式，但 "won't" 是标准写法。

**错误 7：混淆 "be going to" 和 "go to"**
- ❌ "I'm going to the meeting tomorrow."（这句话语法正确，但意思是"我明天要去那个场合开会"——不是"打算"的意思）
- ✅ "I'm going to **attend** the meeting tomorrow."（明确表示"参加"）
- 关键：be going to + 动词原形 = 打算做。be going to + 名词 = 要去某个地方。

## 2. Vocabulary: Top 20 Error & Bug Terms

以下 20 个词汇是你在处理线上事故、写 bug report、参加 incident review 时最常用到的单词。每个单词学习三个常见的语境义项。

---

### 1. crash
- **义项 1: 系统崩溃 (v.)** — 程序或系统突然停止运行
  - "The production server **crashed** at 2 AM, causing a 15-minute outage."
- **义项 2: 崩溃事件 (n.)** — 系统崩溃的实例
  - "We need to analyze the core dump from the last **crash** to identify the root cause."
- **义项 3: 跌落/暴跌 (n.)** — 在非技术语境中，指数值或价格的急剧下降
  - "The stock market **crash** affected our company's valuation."
- **用法提示:** crash 既是动词也是名词。作为动词时，是不及物的（❌ "The server crashed the system." ✅ "The server crashed."）。如果说"导致崩溃"，用 "cause a crash"。

---

### 2. timeout
- **义项 1: 请求超时 (n.)** — 请求在规定时间内没有得到响应
  - "The database query returned a **timeout** after waiting for 30 seconds."
- **义项 2: 超时中断 (v.)** — 因超过时间限制而自动终止
  - "The connection **timed out** because the server was overloaded."
- **义项 3: 超时设置 (n.)** — 程序中设置的超时阈值
  - "We reduced the connection **timeout** from 60 seconds to 10 seconds to fail fast."
- **用法提示:** 作动词时是短语动词 time out，过去式 timed out。作名词时一个词 timeout。常见搭配：request timeout、connection timeout、timeout error。

---

### 3. exception
- **义项 1: 异常 (n.)** — 程序执行中出现的异常情况（编程术语）
  - "The application threw a null pointer **exception** when the user object was undefined."
- **义项 2: 例外情况 (n.)** — 不符合一般规则的情况
  - "As an **exception**, we allowed the hotfix to bypass the usual review process."
- **义项 3: 例外/除外 (n.)** — 不包含在一般情况中的事物
  - "All API endpoints require authentication, with the **exception** of the health check endpoint."
- **用法提示:** "throw an exception" = 抛出异常，"catch an exception" = 捕获异常。在代码 review 中常说：❌ "The function throws a exception."（a 应该用 an）✅ "The function throws **an** exception."

---

### 4. bottleneck
- **义项 1: 性能瓶颈 (n.)** — 系统中限制整体性能的环节
  - "The database query is the main **bottleneck** — it takes 2 seconds per request."
- **义项 2: 流程瓶颈 (n.)** — 工作流程中拖慢进度的环节
  - "Code review is our biggest **bottleneck** — PRs wait an average of 3 days for approval."
- **义项 3: 狭窄通道 (n.)** — 字面意义上狭窄的路段或通道
  - "The network **bottleneck** is between the load balancer and the application servers."
- **用法提示:** bottleneck 一般用作名词，但也可以作动词："The slow query is **bottlenecking** the entire system." 常见搭配：identify/remove/become a bottleneck。

---

### 5. outage
- **义项 1: 服务中断 (n.)** — 系统或服务不可用的一段时间
  - "The **outage** lasted 45 minutes and affected all users in the Asia-Pacific region."
- **义项 2: 停电 (n.)** — 供电中断
  - "The data center experienced a power **outage** last night, triggering the backup generators."
- **义项 3: 中断/停止 (n.)** — 任何形式的服务或供应中断
  - "We scheduled a maintenance **outage** for Sunday at 2 AM."
- **用法提示:** outage 强调"一段不可用的时间"。outage ≠ crash——crash 是"崩溃事件"（原因），outage 是"不可用时间"（结果/状态）。常见搭配：prolonged outage、network outage、power outage。动词短语：go down / experience an outage。

---

### 6. corrupt
- **义项 1: 损坏数据 (v.)** — 使数据变得不可用或不完整
  - "The unexpected shutdown **corrupted** the database index files."
- **义项 2: 损坏的 (adj.)** — 描述数据或文件已经损坏
  - "We detected **corrupt** data in the user profile table and restored it from backup."
- **义项 3: 腐败的 (adj.)** — 在非技术语境中，指道德败坏或贪污
  - "The file system was **corrupted** after the disk write operation was interrupted."
- **用法提示:** corrupt 既是动词也是形容词。名词形式是 corruption（数据损坏/腐败）。注意区分：corrupted data（已损坏的数据）vs. corrupt data（损坏状态的数据）——两者基本同义，但 corrupted 更强调"被弄坏了"的过程。

---

### 7. leak
- **义项 1: 内存泄漏 (n.)** — 程序未能释放已分配的内存
  - "The memory **leak** in the image processing module caused the server to use 16 GB of RAM."
- **义项 2: 数据泄露 (n.)** — 敏感信息被未授权访问或暴露
  - "The data **leak** exposed customer email addresses and hashed passwords."
- **义项 3: 泄漏 (v.)** — 信息或资源非预期地流出
  - "The application **leaks** file handles because it never closes open connections."
- **用法提示:** 技术语境中最常见的是 memory leak（内存泄漏）和 data leak（数据泄露）。区分 leak 和 breach：leak 可能是无意造成的，breach 通常指被攻破。动词过去式 leaked，不是 ❌ leak/leaked 但从不 ❌ lake。

---

### 8. race condition
- **义项 1: 竞态条件 (n.)** — 多个进程或线程因执行顺序不确定而导致的行为异常
  - "The intermittent bug was caused by a **race condition** between two concurrent API calls."
- **义项 2: 竞争问题 (n.)** — 任何因时序问题导致的不可预测行为
  - "We introduced a **race condition** when we added async processing without proper locking."
- **义项 3: 并发缺陷 (n.)** — 多线程/异步环境中的常见 bug 类型
  - "**Race conditions** are notoriously difficult to reproduce because they depend on timing."
- **用法提示:** race condition 是复合名词。它的形容词形式是 race-condition-related。调试竞态条件的常用策略：加锁（locking）、用事务（transaction）、用互斥量（mutex）。

---

### 9. deadlock
- **义项 1: 死锁 (n.)** — 两个或多个进程互相等待对方释放资源，导致都无法继续
  - "The database **deadlock** occurred when two transactions tried to lock the same tables in different orders."
- **义项 2: 僵局 (n.)** — 在非技术语境中，讨论或谈判陷入僵局
  - "The discussion about the architecture reached a **deadlock** — neither side wanted to compromise."
- **义项 3: 死锁状态 (v.)** — 进入无法继续的状态
  - "The application **deadlocked** after the concurrent requests conflicted on the same resource."
- **用法提示:** deadlock 常和以下动词搭配：cause/lead to/result in a deadlock、resolve/break a deadlock。区分 deadlock 和 race condition：deadlock = 卡住了（谁也动不了），race condition = 结果不确定（可能出错但没卡住）。

---

### 10. regression
- **义项 1: 回归 bug (n.)** — 新代码导致之前正常的功能出现问题
  - "The latest release introduced a **regression** in the payment flow — users can't complete checkout."
- **义项 2: 退化 (n.)** — 在非技术语境中，指质量的下降
  - "There's been a **regression** in code quality since the team started rushing features."
- **义项 3: 回归测试 (n.)** — 通过测试确保新代码不破坏已有功能
  - "We should add a **regression** test for this bug to prevent it from happening again."
- **用法提示:** regression 常和 introduce/cause 搭配："introduce a regression" = 引入了回归 bug。"regression test" = 回归测试，是 QA 流程的核心部分。

---

### 11. edge case
- **义项 1: 边界情况 (n.)** — 只在极端或非典型输入下才会出现的问题
  - "The bug only occurs in the **edge case** where the user enters a negative number."
- **义项 2: 特殊情况 (n.)** — 不太常见但需要考虑的场景
  - "We need to handle the **edge case** where the API returns an empty response."
- **义项 3: 极端测试场景 (n.)** — 在测试中需要覆盖的非典型输入组合
  - "Our test suite should cover all the **edge cases**, not just the happy path."
- **用法提示:** edge case vs. corner case：edge case 是在一个输入维度的极端值，corner case 是多个维度的极端值同时出现。在面试中，"考不考虑 edge case" 是考察系统设计能力的重要指标。

---

### 12. breach
- **义项 1: 安全漏洞 (n.)** — 安全系统被突破导致数据或系统被未授权访问
  - "The security **breach** compromised over 1 million user records."
- **义项 2: 突破防御 (v.)** — 攻破安全防护
  - "The attacker **breached** the firewall through an unpatched vulnerability."
- **义项 3: 违反合约 (v./n.)** — 违反协议或合同条款
  - "Selling customer data would be a **breach** of our privacy agreement."
- **用法提示:** 区分 breach 和 leak：breach 强调"被攻破"（有人入侵），leak 强调"泄露"（数据流出——不一定有入侵）。常见搭配：data breach、security breach、breach of contract。

---

### 13. exploit
- **义项 1: 利用漏洞 (v.)** — 利用系统弱点进行攻击
  - "The attacker **exploited** a known vulnerability in the outdated library."
- **义项 2: 漏洞利用程序 (n.)** — 用于攻击的代码或工具
  - "A public **exploit** for this vulnerability was released on GitHub, so we need to patch urgently."
- **义项 3: 充分利用 (v.)** — 在非安全语境中，指充分利用某资源或机会（中性词）
  - "We should **exploit** the caching layer more effectively to reduce database load."
- **用法提示:** 义项 3 是中性词（"充分利用"），在技术讨论中很常见。但在安全语境中通常是负面含义。名词形式是 exploitation。

---

### 14. patch
- **义项 1: 补丁 (n.)** — 修复代码问题的小更新
  - "The security team released a **patch** for the critical vulnerability."
- **义项 2: 打补丁 (v.)** — 应用补丁来修复问题
  - "We need to **patch** all production servers with the latest security update."
- **义项 3: 快速修复 (v.)** — 非正式地用临时方案修复问题
  - "I **patched** the bug with a quick fix, but we need a proper solution later."
- **用法提示:** patch 和 hotfix 的区别：patch 通常是正式的、经过测试的修复；hotfix 是紧急的、小范围的修复。常见搭配：security patch、patch management、patch Tuesday。

---

### 15. hotfix
- **义项 1: 热修复 (n.)** — 紧急发布的修复补丁（不经过完整发布流程）
  - "We had to push a **hotfix** to production at 3 AM to resolve the payment issue."
- **义项 2: 紧急修复 (n.)** — 针对生产事故的紧急响应
  - "The **hotfix** bypassed the usual CI pipeline because it was a critical blocker."
- **义项 3: 热修复程序 (v.)** — 在运行系统中直接应用修复
  - "We **hotfixed** the configuration file on all servers without restarting the service."
- **用法提示:** hotfix 是 compound noun，不可写作 hot fix 或 hot-fix（虽然有些人这样写）。区分 hotfix 和 patch：hotfix 强调"紧急"、"不按常规流程"；patch 更正式。

---

### 16. stale
- **义项 1: 过期的/陈旧的 (adj.)** — 数据或缓存已不再是最新
  - "The dashboard was showing **stale** data because the cache hadn't been refreshed."
- **义项 2: 停滞的 (adj.)** — PR 或分支长时间无人更新
  - "We have 15 **stale** pull requests that haven't been reviewed in over a week."
- **义项 3: 陈旧的代码 (adj.)** — 不再维护或使用的代码
  - "The repository contains a lot of **stale** code from experiments that were never cleaned up."
- **用法提示:** stale 是形容词，名词形式是 staleness。在敏捷团队中，"stale PR" 是一个常见问题。常见搭配：stale data、stale branch、stale cache、go stale。

---

### 17. flaky
- **义项 1: 不稳定的测试 (adj.)** — 测试结果时好时坏，不是确定性失败
  - "We have 10 **flaky** tests that fail intermittently — we need to fix them before the release."
- **义项 2: 不靠谱的 (adj.)** — 描述可能随时出现问题的事物
  - "The third-party API is **flaky** — sometimes it responds in 100ms, other times it times out."
- **义项 3: 易变的 (adj.)** — 在非技术语境中，形容人或事物的不可靠
  - "The network connection in the office is **flaky**, so I often lose VPN access."
- **用法提示:** flaky 是形容词，常用于测试领域："flaky test" 是 QA 团队的最头疼的问题之一。名词形式是 flakiness。"flaky test" 的反面是 "deterministic test"（确定性测试）。

---

### 18. obsolete
- **义项 1: 过时的/废弃的 (adj.)** — 不再被使用或支持
  - "The v1 API endpoint is **obsolete** — all clients should migrate to v2 by next quarter."
- **义项 2: 淘汰的 (adj.)** — 已经被新技术替代
  - "The legacy system runs on an **obsolete** technology stack that nobody on the team knows."
- **义项 3: 作废的 (adj.)** — 不再有效或适用
  - "That documentation is **obsolete** — it describes the old architecture before the microservices migration."
- **用法提示:** obsolete 和 deprecated 的区别：deprecated = 仍然可用但不推荐（短期内还在），obsolete = 已经不再使用/支持了。obsolete 比 deprecated 更"严重"。常见搭配：become obsolete、render obsolete、obsolete technology。

---

### 19. duplicate
- **义项 1: 重复的 (adj.)** — 存在多个相同的内容
  - "We removed **duplicate** records from the database, reducing the table size by 30%."
- **义项 2: 复制/重复 (v.)** — 复制某物或重复某操作
  - "The cron job **duplicated** the data because of a logic error in the script."
- **义项 3: 重复项 (n.)** — 重复或相同的事物
  - "This ticket is a **duplicate** of the one filed yesterday — I'll close it."
- **用法提示:** duplicate 既是动词、名词又是形容词。在 bug tracking 中，"duplicate ticket" 很常见——意思是"跟之前的 ticket 重复了"。常见搭配：duplicate data、duplicate record、eliminate duplicates。发音注意：作动词时读 /ˈduːplɪkeɪt/，作名词/形容词时读 /ˈduːplɪkət/。

---

### 20. inconsistent
- **义项 1: 不一致的 (adj.)** — 数据或行为前后矛盾
  - "The test results are **inconsistent** — the same test passes on one run and fails on the next."
- **义项 2: 不统一的 (adj.)** — 代码风格或实现方式缺乏统一性
  - "The codebase has **inconsistent** naming conventions — some files use camelCase, others use snake_case."
- **义项 3: 不稳定 (adj.)** — 行为不统一或不可预测
  - "The API behavior is **inconsistent** — it sometimes returns an array and sometimes returns null."
- **用法提示:** 名词形式是 inconsistency（不一致）。反义词是 consistent（一致的/稳定的）。在代码 review 中，经常看到："This naming is **inconsistent** with the rest of the codebase." 常见搭配：inconsistent behavior、inconsistent data、inconsistent results。

## 3. Sentence-Making Practice

以下 10 个练习结合了 Month 1 所有时态（一般现在时、现在进行时、一般过去时、过去进行时、will 将来、be going to 将来、现在进行时表将来、一般现在时表将来）和全部词汇。每个练习模拟一个真实工作场景。

---

### 1. 你在 standup 上说：我今天打算修复那个内存泄漏的问题，之前它导致了两次崩溃。用 **be going to** 和词汇 **memory leak**、**crash**。

**参考答案:** "I'm going to fix the memory leak today. It caused two crashes last week."
> （be going to 表达"计划"——你今天计划做什么。It caused two crashes 用一般过去时，因为崩溃已经在上周发生了。注意："memory leak" 前加 the，因为特指"那个"特定的泄漏。）

---

### 2. 你和同事在排查一个问题，告诉他你会去查 timeout 是不是因为数据库瓶颈。用 **will**（当场决定）和词汇 **timeout**、**bottleneck**。

**参考答案:** "I'll check if the timeout is caused by a database bottleneck."
> （will 表达当场决定——你在对话中临时提出的检查计划。is caused 用一般现在时被动语态。"if" 从句里用一般现在时，因为描述的是"当前是否存在"的情况。）

---

### 3. 你告诉团队，这个周五 sprint 结束（固定时间表），而且你们打算在下个 sprint 解决所有的回归 bug。用 **一般现在时表将来** 和 **be going to**，使用词汇 **regression**。

**参考答案:** "The sprint ends on Friday, and we're going to fix all the regressions next sprint."
> （一般现在时 ends 表固定时间表（sprint 结束日期是定好的）。be going to 're going to 表计划好的行动。regression 用复数 regressions，因为可能不止一个。注意：不说 "will fix" 因为这是计划而非当场决定。）

---

### 4. 你在和 QA 确认——你们已经安排好了明天下午一起审查 flaky 测试。用 **现在进行时表将来** 和词汇 **flaky**。

**参考答案:** "We're reviewing the flaky tests tomorrow at 2 PM. Are you available?"
> （现在进行时表将来 're reviewing——说明已经安排好了。tomorrow at 2 PM 是具体时间，让"将来"的含义明确。flaky 修饰 tests，表示"不稳定的测试"。）

---

### 5. 在 incident review 上，你描述昨天的事故：你正在看错误日志的时候，服务器 crash 了。用 **过去进行时 + when + 一般过去时** 和词汇 **crash**。

**参考答案:** "I was reviewing the error logs when the server crashed yesterday."
> （过去进行时 was reviewing 描述"正在查看日志"这个长动作（背景），一般过去时 crashed 描述"崩溃"这个瞬间事件（插入）。crashed 是规则动词，-ed 读 /t/（因为 sh 是清辅音）。）

---

### 6. 你发现数据库有损坏的记录，告诉团队这必须立即处理——不能等到下周。用 **won't** 和词汇 **corrupt**、**hotfix**。

**参考答案:** "We found corrupt records in the database. We won't wait until next week — we need to push a hotfix today."
> （一般过去时 found 描述"发现了"这个过去事件。won't wait 是 will 的否定将来——表达"坚决不"的态度。corrupt 是形容词修饰 records。hotfix 是名词作 push 的宾语。注意：won't 是 will not 的缩写，口语中比 will not 更常见。）

---

### 7. 要解释为什么你们团队选择重构而不是打补丁——因为现在的代码已经过时了，而且还有不一致的命名问题。用 **一般现在时** 和词汇 **obsolete**、**inconsistent**。

**参考答案:** "We chose to refactor instead of patching because the current code is obsolete and the naming conventions are inconsistent."
> （一般过去时 chose 描述已做出的决定。一般现在时 is obsolete 和 are inconsistent 描述当前状态——代码现在就是过时的和不一致的。patching 是动名词，跟在介词 of 后面。注意："chose" 是 choose 的不规则过去式 /tʃoʊz/。）

---

### 8. 你向安全团队报告：昨天的安全漏洞被利用了，因为一个已知的漏洞一直没有打补丁。用 **一般过去时** 和词汇 **breach**、**exploit**、**patch**。

**参考答案:** "The security breach yesterday exploited a known vulnerability that we hadn't patched."
> （一般过去时 exploited 描述"利用"这个事件。that we hadn't patched 是过去完成时（Weeks 3 学过）——描述"在漏洞被利用之前一直没有打补丁"。注意：如果你是本周才学到这里，用 "that we didn't patch" 也可以，因为前后两个动作都在过去——we didn't patch it, then it was exploited。）

---

### 9. 在规划会议上，你预计如果团队继续加新功能，构建时间会显著增加。用 **be going to**（基于证据的预测）和词汇 **edge case**、**regression**。

**参考答案:** "If we keep adding new features without improving our test coverage, the build time is going to increase, and we're going to miss more edge cases, causing regressions."
> （be going to 表达"基于当前趋势的预测"——不是主观猜测，而是基于"不断加新功能"这个事实。if 从句用一般现在时 keeps adding（条件状语从句中不用将来时）。"miss more edge cases" = 漏掉更多边界情况。"causing regressions" 是现在分词短语表结果。）

---

### 10. 你告诉实习生：我们一般每周一下午部署，但这次我们周二再部署，因为周一有个竞态条件问题需要先修复。用 **一般现在时（常规）**、**be going to（计划）** 和词汇 **race condition**。

**参考答案:** "We usually deploy every Monday afternoon, but we're going to deploy on Tuesday this week because we need to fix the race condition first."
> （一般现在时 deploy 描述常规操作（每周一的固定部署）。be going to 're going to 描述本周的特殊计划（因为竞态条件没修好）。need to fix 也是一般现在时——描述"需要修复"这个当前需求。注意：every Monday afternoon 前面的介词 on 可以省略——"every Monday" 本身就是时间状语。）

### A. Tense Overview Table（时态概览表）

这是 Month 1 学过的全部 8 种时态/语法结构的总览。

| 时态 | 公式 | 核心用法 | 一句话示例 |
|------|------|---------|-----------|
| **Simple Present** 一般现在时 | V / V-s/es | 事实、习惯、当前状态 | "We deploy every Monday." |
| **Present Continuous** 现在进行时 | am/is/are + V-ing | 正在进行、临时情况 | "I'm debugging a memory leak." |
| **Simple Past** 一般过去时 | V-ed / 不规则过去式 | 过去完成的动作/事件 | "The server crashed at 2 AM." |
| **Past Continuous** 过去进行时 | was/were + V-ing | 过去某时刻正在进行的动作 | "I was reviewing the PR when you called." |
| **will + 原形** 一般将来时 | will + V | 当场决定、预测、承诺 | "I'll check the logs." |
| **be going to + 原形** 打算将来 | am/is/are + going to + V | 计划、有证据的预测 | "We're going to migrate the DB." |
| **Present Continuous（将来）** 现在进行时表将来 | am/is/are + V-ing + 将来时间 | 已经安排好的日程 | "I'm meeting the PM tomorrow." |
| **Present Simple（将来）** 一般现在时表将来 | V-s/es + 将来时间 | 固定时间表 | "The sprint ends on Friday." |

**时态选择的核心原则：**

```
时态 = 时间 + 状态
 时间：过去 ← 现在 → 将来
 状态：简单（事实/结果）← → 进行（过程/背景）
```

| 时间\状态 | 简单 (Simple) | 进行 (Continuous) |
|-----------|--------------|-------------------|
| **现在** | 事实/习惯 | 正在进行 |
| **过去** | 已完成的事件 | 过去正在进行 |
| **将来** | 当场决定 / 承诺 / 时间表 | 安排好的计划 |

### B. Sentence Pattern Review（句型回顾）

Week 1 学的五种基本句型，用 Month 1 的词汇各举一例。

**Pattern 1: S + V（主语 + 不及物动词）**
- **示例:** "The production server crashed last night."
- 语法分解: The production server (主语/名词短语) + crashed (谓语/不及物动词) + last night (时间状语)
- 说明: crashed 是不及物动词，后面不需要宾语。

**Pattern 2: S + V + O（主语 + 及物动词 + 宾语）**
- **示例:** "We patched the security vulnerability yesterday."
- 语法分解: We (主语/代词) + patched (谓语/及物动词) + the security vulnerability (宾语/名词短语) + yesterday (时间状语)
- 说明: patched 是及物动词——"打补丁"这个动作需要作用在"漏洞"上。

**Pattern 3: S + V + C（主语 + 系动词 + 补足语）**
- **示例:** "The flaky tests are a major blocker for the release."
- 语法分解: The flaky tests (主语/名词短语) + are (系动词) + a major blocker (补足语/名词短语)
- 说明: 系动词 are 连接主语和补足语——"flaky tests = a major blocker"。

**Pattern 4: S + V + O + O（主语 + 动词 + 间接宾语 + 直接宾语）**
- **示例:** "The team lead assigned me the race condition investigation."
- 语法分解: The team lead (主语/名词短语) + assigned (谓语) + me (间接宾语/代词) + the race condition investigation (直接宾语/名词短语)
- 说明: me 是"人"（间接宾语），the investigation 是"事"（直接宾语）。顺序是"人→物"。

**Pattern 5: S + V + O + C（主语 + 动词 + 宾语 + 宾语补足语）**
- **示例:** "The outage kept the service unavailable for two hours."
- 语法分解: The outage (主语/名词) + kept (谓语) + the service (宾语/名词短语) + unavailable (宾语补足语/形容词) + for two hours (时间状语)
- 说明: unavailable 描述宾语 the service 的状态。宾语和补足语之间有隐含的主谓关系：the service (was) unavailable。

### D. Common Pitfalls Summary（Month 1 常见错误 TOP 10）

以下是 Month 1 四个星期中，中国学生最容易犯的 10 个错误。

**1. 忘记第三人称单数的 -s（Week 1-2）**
- ❌ "The API return a JSON object."
- ✅ "The API return**s** a JSON object."
- 原因：中文动词没有形态变化。这是最顽固的错误，没有捷径——只有靠检查。

**2. 现在进行时忘记 be 动词（Week 2）**
- ❌ "I debugging the issue."
- ✅ "I **am** debugging the issue."
- 原因：中文"我正在调试"中的"正在"可以单独作时间副词，但英语的进行时必须用 be 动词。

**3. 状态动词用了进行时（Week 2）**
- ❌ "I am knowing the answer."
- ✅ "I know the answer."
- 原因：状态动词（know, belong, need）描述的是"状态"而非"动作"——状态是已经存在的，不是"正在进行"的。

**4. 双重过去标记（Week 3）**
- ❌ "I didn't **deployed** the fix."（didn't 和 deployed 都标记了"过去"）
- ✅ "I didn't **deploy** the fix."（deploy 用原形）
- 原因：一句话只需要一个过去标记。did/didn't 已经体现了"过去"，后面的动词保持原形。

**5. 不规则动词用成规则变化（Week 3）**
- ❌ "I **writed** a script."（write 不是规则动词）
- ✅ "I **wrote** a script."
- 原因：英语约 200 个不规则动词——写对需要大量练习。

**6. 所有将来都用 will（Week 4）**
- ❌ "I **will meet** the client tomorrow at 3."（听起来像当场决定）
- ✅ "I'm meeting the client tomorrow at 3."（安排好了用现在进行时更自然）
- 原因：中文一个"会"字走天下，但英语的将来表达取决于"确定程度"。

**7. will 后面加 -s 或 -ing（Week 4）**
- ❌ "It **will works**." / "It **will working**."
- ✅ "It **will work**."（will + 动词原形）
- 原因：will 是情态动词，和 can/must/should 一样——后面永远跟动词原形。

**8. 时间/条件状语从句中用 will（Week 4）**
- ❌ "I'll deploy when the tests **will pass**."
- ✅ "I'll deploy when the tests **pass**."
- 原因：在 if, when, until, as soon as 等引导的从句中，即使表示将来，也用一般现在时代替。

**9. be going to 忘记 be 动词（Week 4）**
- ❌ "We **going to** migrate the database."
- ✅ "We **are going to** migrate the database."
- 原因：be going to 中的 be 动词是语法要求，不是可选成分。

**10. 时态混搭——在同一个时间范围内频繁切换时态（全 Month）**
- ❌ "Yesterday, I **deploy** the fix. The server **was crashing** and I **fix** it."
- ✅ "Yesterday, I **deployed** the fix. The server **crashed** and I **fixed** it."
- 原因：如果在描述同一段过去的时间，所有主要动作的时态应该保持一致。不要在同一段话里从过去时切换到现在时又切换回去。

### 易混淆词对比（Month 1 全四章精选）

**1. deploy vs. rollback（部署 vs. 回滚）**
- "We **deployed** v3.0 to production at 10 AM."（向前发布）
- "We **rolled back** to v2.8 at 2 PM because of the memory leak."（向后恢复）
- 关键区别：方向不同——deploy 是"上线"，rollback 是"退回"。

**2. parameter vs. argument（形参 vs. 实参）**
- "The function takes two **parameters**: `userId` and `callback`."（定义时的占位符）
- "When calling the function, pass the correct **arguments**."（调用时传入的实际值）
- 关键区别：parameter = 函数定义的"槽位"，argument = 调用时填入的"值"。

**3. latency vs. throughput（延迟 vs. 吞吐量）**
- "We need to reduce **latency** because every millisecond counts."（响应时间——快慢问题）
- "We need to increase **throughput** to handle more concurrent users."（处理能力——多少问题）
- 关键区别：latency = 一次请求多快（时间维度），throughput = 单位时间能处理多少（容量维度）。

**4. deprecated vs. obsolete（弃用 vs. 过时）**
- "The v1 API is **deprecated** but still works for now."（不推荐但还能用）
- "The v1 API is **obsolete** — it was shut down last month."（已经完全不支持了）
- 关键区别：deprecated = "请迁移走"（还能用，但别再用了），obsolete = "已经没了"。

**5. leak vs. breach（泄漏 vs. 入侵）**
- "The data **leak** exposed customer emails due to a misconfigured S3 bucket."（无意泄露）
- "The security **breach** was caused by an attacker exploiting a known vulnerability."（被攻破）
- 关键区别：leak = 数据流出（可能是无意的），breach = 被入侵（有人攻进来了）。

**6. crash vs. outage（崩溃 vs. 中断）**
- "The server **crashed** at 2 AM, causing a 45-minute **outage**."（崩溃是原因，中断是结果）
- "The **crash** happened because of a memory leak."（崩溃是"事件"，有具体原因）
- "The **outage** affected 10,000 users."（中断是"时间段"，重点在影响范围）
- 关键区别：crash = 崩溃这件事（事件），outage = 服务不可用的一段时间（后果/状态）。

**7. hotfix vs. patch（热修复 vs. 补丁）**
- "We pushed a **hotfix** at 3 AM because the payment system was completely down."（紧急、非常规）
- "We applied the monthly security **patch** during the maintenance window."（常规、有计划）
- 关键区别：hotfix = 紧急情况，不按常规流程；patch = 较正式的有计划更新。

**8. will vs. be going to（当场决定 vs. 已有计划）**
- "I'll investigate that bug."（当场决定——之前没想过，现在主动揽活）
- "I'm going to investigate that bug."（已经有计划——之前已经安排好了）
- 关键区别：will = "好吧我来"（当场决定），be going to = "我本来就要做"（计划好）。

**9. race condition vs. deadlock（竞态 vs. 死锁）**
- "The **race condition** caused the data to be inconsistent because two processes wrote at the same time."（竞争导致不可预测的结果）
- "The **deadlock** froze the system because each process was waiting for the other to release a lock."（互相等待导致卡死）
- 关键区别：race condition = 时序竞争导致结果不确定，deadlock = 循环等待导致谁都无法继续。

**10. edge case vs. regression（边界情况 vs. 回归 bug）**
- "We missed an **edge case** where the input is an empty string."（之前没考虑到的边界场景）
- "The new release introduced a **regression** — the login feature that worked before is now broken."（新代码破坏了之前正常的功能）
- 关键区别：edge case = "之前没想到的场景"，regression = "之前本来好的现在坏了"。

### 周末练习建议

**建议用时：** 周六 1 小时 + 周日 1 小时（Month 1 收尾，值得多花时间）

**周六练习：写一段完整的 incident postmortem**

用英语写一段 8-10 句话的生产事故复盘。要求：
- 至少使用 3 种不同的将来表达（will, be going to, 现在进行时表将来）
- 至少使用 3 种过去时态（一般过去时、过去进行时、一般现在时、现在进行时）
- 至少使用本周的 5 个 Bug 词汇
- 至少使用 Month 1 学过的 3 个其他词汇

**示例（参考框架）：**
> "Yesterday at 3:15 PM, our monitoring system triggered a critical alert — the payment service was returning 502 errors. We immediately checked the logs and found that the database had reached its connection limit. While the SRE team was restarting the database, I investigated the root cause and discovered a memory leak in the connection pool. The leak was caused by a race condition we introduced in last week's deployment. After the service was restored, we held an incident review. We're going to add a regression test for the race condition and set a proper timeout for database connections. The tech lead is presenting the incident report to the stakeholders on Monday. I'll follow up with the QA team to make sure the new tests are in place before the next deployment."

**句子分析：**
- "our monitoring system triggered" → 一般过去时（事件发生）
- "we immediately checked" → 一般过去时（采取行动）
- "the SRE team was restarting" → 过去进行时（背景动作）
- "I investigated and discovered" → 一般过去时（一系列动作）
- "The leak was caused by" → 一般过去时被动（原因分析）
- "we're going to add" → be going to（计划好的改进措施）
- "The tech lead is presenting" → 现在进行时表将来（已安排的会议）
- "I'll follow up" → will（承诺/当场决定）

写完后，逐句检查：
- 时态选择正确吗？为什么用这个时态？
- 不规则动词写对了吗？
- 第三人称单数加 -s 了吗？
- Bug 词汇用对了吗？
